# Decision log

## 2026-07-16 — 메인 화면을 행동 중심 프라이빗 웰스 콘솔로 단순화

Decision: 초기 화면의 가짜 `NEXT`, `0%`, 비활성 행동 카드를 제거하고 계획이 생성된 뒤에만 `다음 자산 단계`, `행동 3개`, `행동 완료 진행률`을 노출한다. 활성 계획은 3열 카드 대신 하나의 세로 실행 원장으로 표시하고, 자산정보 입력은 `가구 자산 → 실행 여건`의 실제 2단계 흐름으로 분리한다. 시각 언어는 따뜻한 페이퍼 배경, 미네랄 그린, 헤어라인과 제한된 그림자를 사용하는 Quiet Private Banking 방향으로 통일한다.

Why: 분석 정보와 비어 있는 대시보드를 먼저 보여주면 사용자가 판단해야 할 항목이 늘고 서비스가 미완성처럼 보인다. 다음 한 단계와 이번 달 행동만 선명하게 보여줘야 WealthCopy의 정체성인 `복제 → 실행 → 완료`가 즉시 이해된다.

Evidence: 320px, 390px, 1440px 뷰포트에서 가로 넘침 없이 초기·입력·활성·완료 화면을 검증했다. 실제 계획 API로 `L6 → L7`과 `L15 → L15` 유지 경로를 생성하고 행동 완료율 `0 → 33 → 67 → 100%` 연결을 확인했다. 전체 60개 테스트, lint, typecheck, production build가 통과했으며 Cloud Run revision `wealth-copy-00002-jmh`가 100% 트래픽을 받고 배포 후 오류 로그는 0건이다.

Limits: 고급화된 화면도 교육용 행동 기록 도구이며 금융 거래, 투자 판단, 목표 단계 도달을 보장하지 않는다. 행동 3개 완료 후에도 최신 자산 스냅샷으로 레벨을 다시 분류해야 한다.

## 2026-07-15 — 서울 Cloud Run에 최소권한 공개 데모 배포

Decision: Next.js를 Node.js 24 standalone 컨테이너로 만들고 GCP 프로젝트 `abis-web-platform`의 서울 리전 Cloud Run 서비스 `wealth-copy`에 배포한다. 런타임은 전용 서비스 계정 `wealth-copy-run`을 사용하고 OpenAI 키는 Secret Manager의 버전 고정 환경변수로만 주입한다. 공개 데모는 최소 인스턴스 0, 최대 3, 동시 요청 20, 요청 제한 120초로 운영한다.

Why: 메인 화면과 서버 계획 API를 같은 origin에서 운영하면서 TLS, scale-to-zero, revision과 로그를 관리형 서비스로 확보할 수 있다. 전용 서비스 계정과 secret 단위 접근 권한은 다른 프로젝트 리소스와 OpenAI 키의 권한 범위를 줄인다.

Evidence: revision `wealth-copy-00001-lkh`가 100% 트래픽을 받고 있으며 공개 페이지, 실제 OpenAI 요청, `L1 → L2`, `L14 → L15`, `L15 → L15` API 계약과 no-store 헤더가 검증됐다. 배포 후 Cloud Run 오류 로그와 OpenAI 오류 로그는 0건이었다.

Limits: 현재 IP·세션 rate limit은 인스턴스 메모리에만 있어 scale-out 전체에 대한 전역 제한이 아니다. 정식 운영 전 인증, 분산 rate limit, 비용 알림, 보존·삭제 정책, 모니터링과 법률 검토가 필요하다. Cloud Run 기본 TCP startup probe를 사용하며 별도 애플리케이션 health endpoint는 아직 없다.

## 2026-07-15 — 가구 순자산으로 L1–L15를 자동분류

Decision: 사용자가 입력한 `가구 총자산`과 `가구 총부채`로 `가구 순자산 = 총자산 - 총부채`를 계산하고 `krw-net-worth-v1` 제품 정책에 따라 현재 레벨을 자동분류한다. 클라이언트가 현재 레벨을 직접 지정하지 않는다.

| Level | 가구 순자산 |
| --- | ---: |
| `L1` | 0원 미만 |
| `L2` | 0원 이상 1천만원 미만 |
| `L3` | 1천만원 이상 3천만원 미만 |
| `L4` | 3천만원 이상 1억원 미만 |
| `L5` | 1억원 이상 3억원 미만 |
| `L6` | 3억원 이상 5억원 미만 |
| `L7` | 5억원 이상 10억원 미만 |
| `L8` | 10억원 이상 30억원 미만 |
| `L9` | 30억원 이상 50억원 미만 |
| `L10` | 50억원 이상 100억원 미만 |
| `L11` | 100억원 이상 300억원 미만 |
| `L12` | 300억원 이상 1,000억원 미만 |
| `L13` | 1,000억원 이상 3,000억원 미만 |
| `L14` | 3,000억원 이상 1조원 미만 |
| `L15` | 1조원 이상 |

Why: 자가선택 단계는 사용자마다 해석이 달라 경로의 출발점이 일관되지 않았다. 하나의 버전형 순자산 정책으로 분류해야 모든 사용자가 동일한 경계와 다음 단계 규칙을 사용한다.

Evidence: 정책 모듈은 총자산·총부채를 0 이상의 safe integer로 제한하고, 모든 정확한 하한과 하한 직전 값, 음의 순자산, 부채 차감 사례를 테스트한다. 모든 하한은 포함하고 상한은 제외한다.

Limits: 이 레벨은 WealthCopy가 행동을 구성하기 위한 제품 구간이다. 공식 자산등급, 한국 자산 백분위, 신용등급, 투자 적합성 또는 미래 성과가 아니다. 경계 변경에는 새 정책 버전과 저장 마이그레이션이 필요하다.

## 2026-07-15 — L15는 최상위 유지 단계

Decision: `L1 → L2`부터 `L14 → L15`까지 한 단계씩 연결하고, `L15`는 더 높은 목표를 만들지 않고 `L15 → L15` 유지·운영 점검 행동을 제공한다.

Why: 1조원 이상 구간에서 존재하지 않는 다음 등급을 제시하면 제품 계약과 사용자 카피가 거짓이 된다. 유지 단계도 매월 기록·권한·연속성을 점검할 실질적 행동이 필요하다.

Limits: L15 행동은 수익 확대, 투자 배분, 세금·법률 결론 또는 거래를 제안하지 않는다.

## 2026-07-15 — 정확한 금융 금액을 분류 경계에 격리

Decision: 정확한 총자산·총부채는 요청에서 현재 레벨을 계산하는 동안만 사용한다. 금액을 OpenAI 모델 입력, 성공 응답, 브라우저 저장소, ICS 파일, 분석 이벤트 또는 애플리케이션 로그에 포함하지 않는다. API 응답은 `Cache-Control: no-store`, 모델 호출은 `store: false`를 사용한다.

Why: 다음 행동을 만들기 위해 필요한 것은 분류된 단계와 통화 중립 행동 신호이지 정확한 가구 금액이 아니다. 민감정보의 이동과 보존을 최소화해야 한다.

Evidence: 수집 프로필은 서버 경계에서 분류한 뒤 `toNormalizedProfile`로 금액을 제거한다. 공개 계획은 strict 스키마로 다시 투영되고 월말 ICS는 일반적인 행동 점검 일정만 포함한다.

Limits: 계산된 다음 단계와 행동 완료 기록은 기기 내 저장 대상이므로 정확한 금액이 없더라도 금융 맥락을 드러낼 수 있다. 사용자에게 이를 고지하고 삭제 기능을 제공한다.

## 2026-07-15 — PSID 참고값과 원화 레벨 정책을 분리

Decision: `psid-wealth-reference-v1`은 공개 PSID 2019 가족 순자산 집계의 백분위 경계 구조를 선택형 참고값으로만 제공한다. `krw-net-worth-v1`은 원화 가구 순자산으로 L1–L15를 분류하며 PSID 값을 사용하지 않는다. `behavior-policy-v1`은 소득 대비 실행 비율과 부채 부담으로 행동 우선순위를 정한다.

Why: 미국 달러 순자산 분포를 한국 원화 경계로 환산하거나 개인의 한국 백분위로 해석하면 거짓 정밀도가 생긴다. 관측 자료, 제품 분류, 행동 정책을 분리해야 한다.

Evidence: PSID 구간을 바꾸어도 현재 레벨과 실행 속도 점수는 달라지지 않는다. 모델에는 PSID 원 임계값과 정확한 개인 금액을 전달하지 않는다.

Limits: 사용자가 고른 참고 구간은 실제 백분위, 공식 등급, 투자 성향, 위험 감수 능력 또는 단계 도달 가능성을 의미하지 않는다. Public Use 마이크로데이터, Restricted Data와 개별 가족 기록은 사용하지 않는다.

## 2026-07-15 — 3/3 완료와 자산 단계 도달을 분리

Decision: `3/3 · 100%`는 이번 달 세 행동을 모두 체크했다는 뜻만 가진다. 행동 완료로 현재 레벨을 변경하거나 과거 `nextLevel`을 새 현재 레벨로 승계하지 않는다. 새 달에는 항상 최신 자산 스냅샷으로 재분류한다.

Why: 체크리스트 완료는 순자산 임계값 통과의 증거가 아니다. 특히 상위 구간에서 월간 행동과 자산 경계 사이에 인과관계를 주장할 근거가 없다.

Evidence: 완료 화면과 라이브 상태 메시지는 자동 승급을 부정하고 최신 자산정보 재입력을 안내한다. 진행률은 완료 행동 수에서만 결정된다.

Limits: 같은 달에 동일한 출발 단계와 다음 단계로 다시 생성된 계획만 동일 action ID의 완료 상태를 이어받을 수 있다.

## 2026-07-15 — 자동분류 의미 변경에 따라 저장소를 v4로 교체

Decision: 브라우저 키를 `wealthcopy-public-plan-v4`로 올리고 정확히 `{version, monthKey, sourceLevel, plan}`만 저장한다. 별도 정책 식별자 필드를 추가하지 않고 v4 스키마의 의미를 `krw-net-worth-v1` 자동분류에 고정한다. 익명 session UUID는 금융정보 없이 `wealthcopy-anonymous-session`이라는 별도 localStorage 키에 둔다. 과거 `wealthcopy-public-plan-v3`, `wealthcopy-public-plan-v2`, `wealthcopy-demo-plan-v1`은 새 레벨로 변환하지 않고 제거한다.

Why: 과거 단계는 사용자가 고른 행동여정 또는 고정 데모 목표였으므로 새 순자산 구간과 문자열이 같아도 의미가 다르다. 이를 자동 마이그레이션하면 검증되지 않은 금융 분류가 된다.

Evidence: 이전 키가 발견되면 행동과 완료 상태를 승계하지 않고 최신 가구 자산정보로 다시 시작하라는 메시지를 표시한다.

## 2026-07-15 — 단계 전환 앵커·모델 동반 행동·월말 점검으로 세 행동 구성

Decision: 모든 L1–L15 출발 단계는 하나의 검토된 전환 앵커를 가진다. 안전 제약을 우선 적용하고, GPT‑5.6은 해당 단계의 routine allowlist 안에서 동반 행동만 고를 수 있다. `schedule_monthly_checkin`은 항상 세 번째 행동이다.

Why: 단계별 앵커가 실제 경로 차이를 만들고, 제한된 모델 선택은 정성적 상황을 반영하며, 고정 월말 점검은 다음 달의 새 스냅샷과 연결한다.

Limits: 모델은 사용자 카피, 금액, 순자산, 현재·다음 레벨, 수익률, 상품, 거래 또는 진행률을 만들거나 볼 수 없다. 모델이 없거나 출력이 유효하지 않아도 같은 공개 형태의 규칙 fallback을 반환한다.

## 2026-07-15 — 상위 레벨은 운영·거버넌스 행동으로 현실화

Decision: 상위 레벨 행동은 평가 기준일, 부채·보증 기록, 소유·관리 구조, 보고 일정, 의사결정 권한, 업무 연속성과 접근 절차를 점검한다.

Why: 높은 순자산 구간에서 단순한 저축 확대 행동이나 투자 상품 추천은 현실성이 낮고 금융 자문 위험이 높다. 기록과 운영 통제는 금액·수익률 없이도 완료 기준을 만들 수 있다.

Limits: 비밀번호, 인증수단, 계좌정보 또는 비밀정보를 기록·공유하도록 안내하지 않는다. 행동 체크는 자산 평가, 권한 부여, 소유권 변경 또는 거래 실행이 아니다.

## 2026-07-14 — 자산관리를 행동 문제로 정의

Decision: WealthCopy의 사용자 가치는 더 많은 분석이 아니라 이번 달 실행할 행동을 세 개로 줄이는 것이다.

Why: 정보와 비교가 늘어날수록 실행 부담도 커진다. 사용자가 다음 행동을 바로 확인하고 체크할 수 있어야 월간 습관으로 이어진다.

Alternatives: 경로 비교, 추천 이유, 금액·기간·수익 대시보드를 전면에 두는 방식은 제외한다.

## 2026-07-14 — 엄격한 공개 API 투영

Decision: `POST /api/v2/plan` 성공 JSON 본문의 최상위는 정확히 `{nextLevel, actions, progress}`이며 action은 정확히 `{id, completed}`이다. actions는 중복 없는 허용 ID 세 개다. 서버가 분류한 현재 레벨은 본문이 아니라 `X-WealthCopy-Source-Level` 응답 헤더에만 담아 UI가 출발·다음 단계의 연속성을 검증하게 하며, 응답 전체에 `Cache-Control: no-store`를 적용한다.

Why: 내부 분류와 분석이 클라이언트 계약으로 새어 나오지 않고 사용자 경험의 단순성이 데이터 경계에서도 유지된다.

Evidence: 현재 레벨, 금액, PSID 값, 경로, 모델, 출처, 점수와 설명을 성공 JSON 본문에서 제외하고 최종 결과를 strict Zod 스키마로 검증한다. 현재 레벨은 검증용 응답 헤더에만 존재한다.

## 2026-07-14 — Structured Outputs와 동일 형태 fallback

Decision: Responses API, `gpt-5.6`, Zod Structured Outputs와 의미 검증을 사용한다. 키 누락, API 오류, 검증 실패와 rate limit 상황에서도 fallback은 같은 공개 계약을 반환한다.

Why: AI 상태와 무관하게 행동 화면이 동일하며 사용자는 내부 공급자 상태를 이해할 필요가 없다.

## 2026-07-14 — 입력·남용·금융 안전 경계

Decision: 개인정보 형태와 금융 실행·소득 중단 조건을 모델 전에 차단하고, 8KB 본문 제한과 IP·세션 인메모리 rate limit을 적용한다. 어떤 행동도 실제 거래를 실행하지 않는다.
