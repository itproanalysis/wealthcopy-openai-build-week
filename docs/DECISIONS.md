# Decision log

## 2026-07-14 — L6→L7로 MVP 범위 고정

Decision: 첫 데모는 현재 자산그룹 L6에서 목표 자산그룹 L7로 이동하는 한 가지 여정에 집중하고, 스키마에서도 `L6`와 `L7` 리터럴만 허용하며 안정형·균형형·빠른형 세 경로만 비교한다. 순자산은 입력·UI·API에서 수집하지 않는다.

Why: `Compare → Copy → Follow → Grow`의 핵심 가치를 90초 안에 끝까지 보여 줄 수 있고, 범용 자산관리 서비스처럼 과도하게 약속하지 않는다.

Alternatives: 모든 자산그룹 조합, 금융계좌 연동, 상품 추천, 자동 리밸런싱은 검증·규제·통합 범위가 커서 MVP에서 제외했다.

Evidence or follow-up: L6→L7 기준 대표 시나리오는 안정형 77개월·월 240만 원, 균형형 62개월·월 310만 원, 빠른형 54개월·월 420만 원이다. 모두 기획용 추정치로 표시한다.

## 2026-07-14 — 단일 Next.js 저장소

Decision: Next.js App Router, React, strict TypeScript, Tailwind CSS, pnpm, Zod, Vitest를 사용한다.

Why: 하나의 저장소에서 반응형 UI, 서버 전용 API 경계, 입력 검증, 테스트와 프로덕션 빌드를 빠르게 반복할 수 있다.

Alternatives: 별도 백엔드와 데이터베이스는 현재 데모에 필요한 지속 데이터가 없어 통합 비용만 늘린다.

## 2026-07-14 — 수치 결정과 언어 해석 분리

Decision: `src/lib/wealth/engine.ts`가 금액, 기간, 부담 가능 여부, 경로 점수를 결정한다. GPT‑5.6은 이미 계산된 후보의 정성적 적합 이유와 트레이드오프, 확인 질문, 승인된 체크리스트 순서만 구조화해 반환한다.

Why: 고위험 금융 맥락에서 모델이 숫자나 상품을 즉흥 생성하지 못하게 하면서도, 사용자의 자연어 제약을 비교 가능한 설명으로 바꾸는 핵심 AI 가치를 유지한다.

Alternatives: 모델이 계획 전체를 자유 생성하는 방식은 설명은 풍부하지만 숫자 일관성, 재현성, 안전성 검증이 어렵다. 완전 규칙 기반은 자연어 제약 해석이 약하다.

Evidence or follow-up: 엔진과 fallback은 단위 테스트로 고정한다. 향후 실제 사용자 연구가 생기면 경로 라이브러리 자체를 검증하고 버전 관리해야 한다.

## 2026-07-14 — Responses API와 Structured Outputs

Decision: 서버에서 `gpt-5.6`, Responses API, `responses.parse`, Zod Structured Outputs, medium reasoning을 사용한다.

Why: 허용된 경로 ID·상태·이유 코드·근거 필드·트레이드오프 코드·액션 ID만 받으면 UI가 자유 형식 문장을 파싱하지 않아도 되고, 후속 의미 검증이 가능하다.

Alternatives: Chat Completions 또는 자유 형식 텍스트는 현재 프로젝트의 구조화·검증 요구에 비해 추가 파싱과 실패 처리가 필요하다.

Evidence or follow-up: 출력은 정확히 세 경로를 중복 없이 포함해야 하며, `ready`가 아니면 lead 경로가 `null`이어야 한다. 검증 실패 시 모델 출력은 폐기한다.

## 2026-07-14 — 오프라인 우선 fallback

Decision: 클라이언트가 규칙 기반 결과를 먼저 표시하고 `/api/paths/compare`의 AI 해석으로 보강한다. 키 누락, API 오류, 스키마 오류, 의미 오류가 발생하면 fallback 응답과 경고를 반환한다. 실행성 금융 요청은 `professional_review_required`, 소득 중단·연체 등은 `needs_more_information`으로 모델 호출 전에 결정하며 경로 복사를 막는다.

Why: 네트워크와 API 키 상태가 Build Week 데모의 핵심 여정을 막지 않으며, 사용자는 결과의 출처를 화면에서 구분할 수 있다.

Alternatives: AI 실패를 전체 오류 화면으로 처리하면 데모와 사용자 흐름이 중단된다. 실패를 숨기면 결과 출처가 불투명해진다.

## 2026-07-14 — 요청 입력과 남용 방어

Decision: `constraintNote`의 이메일·전화번호·주민등록번호 형태를 클라이언트와 서버에서 거절하고, 본문을 8KB로 제한한다. 모델 호출 경로에는 프로세스 메모리 기반으로 IP당 분당 20회, 익명 세션당 분당 8회 제한을 적용한다.

Why: 불필요한 개인정보가 모델에 전송되는 것을 줄이고, 실수나 반복 호출이 데모 API 비용과 가용성에 미치는 영향을 제한한다.

Alternatives: 클라이언트 검사만으로는 직접 API 호출을 막을 수 없다. 현재 인메모리 제한은 여러 인스턴스에 공유되지 않으므로 공개 서비스의 최종 방어로 간주하지 않는다.

Evidence or follow-up: 제한 초과 응답은 `Retry-After`가 포함된 `429`와 fallback 본문을 제공한다. 공개 출시 전에는 인증 사용자와 IP를 함께 고려하는 분산 rate limiting으로 교체한다.

## 2026-07-14 — 서버 비저장과 브라우저 로컬 계획 저장

Decision: 모델 입력은 월 가용액, 부채 비율, 가구 유형, 선호 경로, 비상자금과 검사된 추가 제약으로 최소화한다. 월소득은 서버 검증에만 사용하고 모델에는 보내지 않는다. 익명 브라우저 세션 UUID는 서버에서 SHA-256 해시해 `safety_identifier`로 보내며 OpenAI 요청은 `store: false`를 사용한다.

Why: 경로 해석에 필요하지 않은 금융정보 전송을 줄이고, 원시 세션 식별자를 모델 API에 전달하지 않는다.

Alternatives: 전체 프로필을 전송하면 구현은 단순하지만 모델 역할에 불필요한 민감 데이터를 노출한다.

Evidence or follow-up: 경로를 확인한 뒤 `wealthcopy-demo-plan-v1`에는 스키마 검증된 프로필, 선택 경로, 체크 상태, 리마인더 화면 상태, 버전을 저장하고 재방문 시 다시 검증해 복원한다. 서버 계정 저장은 없지만 로컬 프로필에도 금융 계획 정보가 포함되므로 공개 출시 전 명시적 동의, 삭제 UX, 개인정보·보존 정책이 필요하다.

## 2026-07-14 — 사용자 선택과 금융 안전 경계

Decision: 모델의 lead는 `추천 실행`이 아니라 `먼저 비교할 경로`로 표현한다. 월 가용액을 초과했거나 assessment가 `ready`가 아닌 경로는 복사할 수 없다. 사용자는 확인 모달의 교육용 안내 체크박스를 먼저 선택해야 계획 복사를 확정할 수 있다.

Why: WealthCopy의 `복사`는 금융 거래가 아니라 교육용 월간 체크리스트 생성이며, 실제 의사결정권은 사용자에게 남겨야 한다.

Alternatives: 자동 선택, 계좌 연동, 주문, 자동이체, 상품·수익률 추천은 MVP의 안전·법률 범위를 벗어난다.

Evidence or follow-up: 리마인더는 기본 꺼짐인 화면 상태 데모이며 실제 알림을 예약하지 않는다. 실제 공개 출시 전 인증, 분산 rate limiting, 모니터링, 접근성, 개인정보, 금융 규제 검토와 전문가 자문이 필요하다.
