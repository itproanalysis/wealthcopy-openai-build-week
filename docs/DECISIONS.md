# Decision log

## 2026-07-15 — 3/3 완료 뒤 상태를 재확인하고 다음 전환을 생성

Decision: `3/3 · 100%`는 이번 달 행동 완료만 뜻한다. 완료 뒤에는 기존 `nextLevel`을 현재 단계 후보로 불러오되 자동 승급시키지 않고, 사용자가 실제 상태를 다시 확인·수정한 다음 그 단계에서 이어지는 새 전환을 생성한다.

Why: 체크리스트 완료만으로 자산 상태가 달라졌다고 주장할 근거가 없다. 재확인 절차를 두면 과장된 성과 표현을 막으면서도 한 달 계획이 다음 여정으로 자연스럽게 연결된다.

Evidence: 현재 월의 완료 계획과 이전 월의 완료 계획 모두 다음 단계 후보를 설정 화면에 제안하며, 사용자가 확정하기 전 공개 계획을 만들지 않는다. 미완료 월간 계획은 같은 행동을 미완료 상태로 이어받는다.

## 2026-07-15 — 단계 전환 앵커·안전한 모델 동반 행동·월말 점검으로 세 행동 구성

Decision: 각 `L1–L7` 출발 단계는 하나의 검토된 전환 앵커 행동을 가진다. GPT‑5.6은 해당 전환의 routine allowlist 안에서 안전한 동반 행동만 고를 수 있고, 안전 제약은 모델 선택보다 우선한다. `schedule_monthly_checkin`은 항상 세 번째 행동이다.

Why: 전환 앵커가 전체군을 실질적으로 구분하고, 제한된 모델 선택은 입력 상황에 맞는 차이를 만들며, 고정 월말 점검은 반복 실행을 다음 달과 연결한다.

Limits: 모델은 카피, 금액, 수익률, 상품, 거래, 진행률을 만들지 않는다. 모델이 없거나 결과가 유효하지 않아도 같은 단계 앵커와 월말 점검을 포함한 규칙 기반 계획을 반환한다.

## 2026-07-15 — 출발 단계를 포함하는 v3 저장과 개인정보 없는 캘린더 연결

Decision: 브라우저 키를 `wealthcopy-public-plan-v3`로 올리고 정확히 `{version, monthKey, sourceLevel, plan}`만 저장한다. 월말 점검은 금융 입력을 포함하지 않는 클라이언트 생성 `.ics` 일정 파일로 연결한다.

Why: `sourceLevel`이 있어야 같은 목표로 보이는 `L6 → L7` 전환과 `L7 → L7` 유지를 구분하고 완료 상태를 올바르게 이어받을 수 있다. ICS는 계좌 연결이나 알림 권한 없이도 행동을 실제 캘린더에 옮기는 가벼운 실행 연결이다.

Limits: `sourceLevel`은 사용자가 선택한 제품 여정 맥락일 뿐 자산 증빙이 아니다. 소득 대비 비율, PSID 참고 구간, 부채비율, 메모, 내부 경로와 모델 출력은 저장하거나 일정에 넣지 않는다.

Migration: `wealthcopy-public-plan-v2`는 과거 고정 대상 데모의 출발 단계 `L6`로 한 번 변환한 뒤 제거한다. 아래 2026-07-14의 v2 저장 결정을 대체한다.

## 2026-07-15 — L7 고정을 해제하고 전체 L1–L7 전환을 제품화

Decision: `L7`은 데모 예시이자 최상위 유지 단계일 뿐이다. 사용자가 직접 고른 현재 WealthCopy 실행 여정에서 `L1 → L2`부터 `L6 → L7`, `L7 → L7 유지`까지 모든 전환에 검토된 월간 행동 경로를 제공한다. 사용자용 이름은 `L1 시작`, `L2 흐름 정리`, `L3 현금 안전망`, `L4 납부 안정`, `L5 월 실행`, `L6 자산 구조`, `L7 장기 유지`로 통일한다. 메인 화면은 계산된 다음 단계, 행동 세 개, 행동 완료율만 표시한다.

Why: 하나의 예시 전환을 제품 전체 규칙으로 고정하면 낮은 단계 사용자는 시작할 수 없고, 상위 단계 사용자는 반복할 이유가 없다. 전체군 전환이 서로 다른 행동으로 이어져야 “경로 복제”가 실제 서비스 기능이 된다.

Limits: 단계는 공식 자산등급이나 한국 자산 백분위가 아니며 PSID 구간에서 산출하지 않는다. 완료율도 다음 단계 도달률이 아니라 이번 달 세 행동의 체크 상태다.

Supersedes: 아래 2026-07-14 기록 중 `L7`을 모든 사용자에게 고정한 범위만 대체한다. 세 행동과 완료율 공개 원칙은 유지한다.

## 2026-07-15 — PSID 참고 자료와 행동 정책을 분리 버전으로 관리

Decision: `psid-wealth-reference-v1`은 공개 PSID 2019 집계표의 출처·백분위 경계 구조·통화 중립 투영만 담당하고, `behavior-policy-v1`은 소득 대비 실행 비율과 부채비율로 내부 경로 점수와 행동 우선순위를 정한다. 자기선택 자산 구간은 실행 속도 점수에서 제외한다.

Why: 순자산 분포에서 높은 위치에 있다는 사실은 투자 성향, 실행 속도, 미래 성공 가능성을 입증하지 않는다. 데이터 관측과 제품 행동 정책을 분리해야 PSID를 근거로 공격적 경로를 추천하는 것처럼 오해되지 않는다.

Evidence: 모든 자산 구간에서 같은 실행 비율·부채비율을 주면 경로 점수가 동일한 테스트와 19/20·39/40 정책 경계 테스트를 둔다. 하위 자산 구간은 현금 여유 확인을 누락하지 않는 안전 제약에만 사용한다.

## 2026-07-15 — 경로 우선순위와 월간 완료 상태를 끝까지 연결

Decision: 단계 전환의 `actionPriority`를 규칙 fallback과 모델 결과 병합에 연결하고, 월말 점검은 항상 세 번째 행동으로 보존한다. 같은 출발 단계에서 행동을 재복제할 때 같은 action ID의 완료 상태만 유지하며, 미완료 계획의 월이 바뀌면 행동을 이어받고 완료 상태를 초기화한다.

Why: 경로 라이브러리가 실제 행동에 영향을 줘야 “경로 복제”가 기술적 사실이 된다. 동시에 사용자가 이미 끝낸 동일 행동을 다시 하게 만들지 않고, 월간 반복은 새 체크 상태로 시작해야 한다.

Evidence: 경로 기반 fallback, 필수 행동 보존, 월말 점검 순서, 재생성 교집합, 월 전환 초기화를 각각 테스트한다. 저장소에는 출발 단계와 공개 계획만 남긴다.

## 2026-07-14 — 공개 PSID 표를 통화 중립 참고 구간으로 사용

Decision: 2019년 미국 가족 순자산에 대한 PSID 가중 집계 공개표의 25·50·75·90백분위 경계 구조만 자산 위치 입력에 사용한다. 사용자가 구간을 직접 고르며 서비스는 개인 자산을 PSID와 대조하거나 백분위를 산출하지 않는다. 소득 대비 실행 비율과 월 부채상환 비율도 퍼센트로 직접 받으며 모든 내부 후보 계산과 모델 입력에서 통화 금액을 제거한다.

Why: 미국 달러 임계값을 원화로 환산하면 물가, 가구 구성과 국가별 자산 분포 차이를 무시한 거짓 정밀도가 생긴다. 넓은 자기선택 구간은 한국 사용자도 개념을 이해할 수 있게 하면서 실제 한국 순위라고 오인하는 것을 막는다.

Evidence: 현재 서버 import 경로의 출처 모듈은 공개 집계표의 기준연도·측정치·원 수치를 감사용으로 기록하지만 모델에는 통화 중립 밴드만 전달한다. 성공 API와 브라우저 저장에는 PSID 필드나 입력 프로필이 포함되지 않으며 프로덕션 클라이언트 번들에서도 원 수치를 검사한다.

Limits: 이 구간은 한국 인구의 실제 자산 백분위, 투자수익률, 미래 자산가치 또는 다음 단계 도달 가능성을 뜻하지 않는다. WealthCopy 행동 단계는 PSID와 무관한 제품 분류다. 등록과 이용조건 동의가 필요한 Public Use 마이크로데이터, 별도 계약 대상인 Restricted Data, 개별 가족 기록은 사용하지 않는다.

Alternatives: 환율 환산, 미국 달러 임계값 노출, 한국 공식 통계처럼 표현하는 방식은 제외했다.

## 2026-07-14 — 자산관리를 행동 문제로 정의

Decision: WealthCopy의 사용자 가치는 더 많은 분석이 아니라 이번 달 실행할 행동을 세 개로 줄이는 것이다.

Why: 정보와 비교가 늘어날수록 실행 부담도 커진다. 사용자가 다음 행동을 바로 체크할 수 있어야 월간 습관으로 이어진다.

Alternatives: 세 경로 비교, 추천 이유, 숫자 대시보드를 전면에 두는 방식은 분석 중심 경험을 반복하므로 제외했다.

## 2026-07-14 — 사용자 화면을 세 요소로 제한

Decision: 메인 화면에는 사용자가 고른 현재 단계에서 계산한 다음 자산 단계, 정확히 세 행동, 행동 완료율만 표시한다.

Why: `어디로 가는가`, `지금 무엇을 하는가`, `얼마나 실행했는가`만으로 월간 행동에 충분하다.

Evidence: 경로, 모델, 금액, 기간, 적합도, 트레이드오프, 추천, 리마인더와 추가 KPI는 사용자 화면과 DOM에서 제외한다.

## 2026-07-14 — 완료율은 행동 상태만 반영

Decision: 완료 행동 수를 `0 → 0%`, `1 → 33%`, `2 → 67%`, `3 → 100%`로 변환한다.

Why: 계좌 연결이 없는 MVP에서 자산 성장률이나 다음 단계 도달률을 표시하면 근거 없는 성과 주장이 된다.

Alternatives: 예상 도달, 남은 금액, 속도 개선, 자산 차트는 사용하지 않는다.

## 2026-07-14 — 엄격한 공개 API 투영

Decision: `POST /api/v2/plan` 성공 응답의 최상위는 정확히 `{nextLevel, actions, progress}`이며 action은 정확히 `{id, completed}`이다. actions는 중복 없는 허용 ID 세 개다.

Why: 내부 분석이 클라이언트 계약으로 새어 나오지 않고 사용자 경험의 단순성이 데이터 경계에서도 유지된다.

Evidence: `paths`, `assessment`, `model`, `source`, 수치, 설명, 점수는 성공 응답에 포함하지 않는다. 최종 응답은 strict Zod 스키마로 검증한다.

## 2026-07-14 — 분석·모델·경로 비교를 서버 내부에 격리

Decision: 규칙 엔진과 경로 라이브러리는 내부 후보를 계산하고 GPT‑5.6은 승인된 행동 ID를 선택·정렬한다. 브라우저는 공개 계획만 받는다.

Why: 모델의 정성적 해석은 활용하면서 금융 숫자와 자유 형식 조언이 사용자 화면에 나타나는 것을 차단한다.

Alternatives: 자유 형식 계획 생성과 클라이언트의 모델 응답 파싱은 재현성·안전성·계약 안정성이 낮아 제외했다.

## 2026-07-14 — Structured Outputs와 동일 형태 fallback

Decision: Responses API, `gpt-5.6`, Zod Structured Outputs와 의미 검증을 사용한다. 키 누락, API 오류, 검증 실패 시에도 fallback은 같은 세 키와 세 행동을 반환한다.

Why: AI 상태와 무관하게 행동 화면이 동일하며 사용자는 내부 공급자 상태를 이해할 필요가 없다.

Evidence: 모델은 숫자·진행률·상품·거래를 만들지 못하고 사용자 화면에 모델명이나 fallback 출처를 표시하지 않는다.

## 2026-07-14 — 월 단위 브라우저 저장

Decision: `wealthcopy-public-plan-v2`에 버전, 월 키와 공개 계획만 저장한다. 월이 바뀌면 행동은 유지하되 완료 상태를 초기화한다.

Why: 입력 프로필이나 내부 분석을 보존하지 않고도 월간 행동 습관을 이어갈 수 있다.

Evidence: 이전 `wealthcopy-demo-plan-v1`은 세 행동 상태로 마이그레이션한 뒤 제거한다.

Superseded: 2026-07-15의 `wealthcopy-public-plan-v3` 결정이 출발 단계 보존과 완료 후 상태 재확인 규칙을 추가해 이 결정을 대체한다.

## 2026-07-14 — 입력·남용·금융 안전 경계

Decision: 개인정보 형태와 금융 실행·소득 중단 조건을 모델 전에 차단하고, 8KB 본문 제한과 IP/세션 인메모리 rate limit을 적용한다. 어떤 행동도 실제 거래를 실행하지 않는다.

Why: 교육용 행동 기록 범위를 지키고 불필요한 개인정보 전송과 반복 호출을 줄인다.

Follow-up: 공개 출시 전 인증 기반 분산 rate limiting, 모니터링, 접근성, 개인정보·보존 정책과 금융 법률 검토가 필요하다.

## 2026-07-14 — 단일 Next.js 저장소

Decision: Next.js App Router, React, strict TypeScript, Tailwind CSS, pnpm, Zod, Vitest와 공식 OpenAI JavaScript SDK를 사용한다.

Why: 사용자 UI, 서버 전용 모델 경계, 엄격한 공개 계약과 테스트를 한 저장소에서 빠르게 반복할 수 있다.
