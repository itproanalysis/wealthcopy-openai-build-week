# Decision log

> 현재 공개 계약은 `wealth-report-v2`입니다. 아래 기록은 시간순 결정 로그이므로 이전 항목의 `wealth-report-v1`은 당시 상태를 설명하며, 위쪽의 최신 결정이 이를 대체합니다.

## 2026-07-20 — 영문 Judge Mode·공식 한국 맥락·사용자 소유 스냅샷

Decision: 공개 API와 한국어 종합 리포트는 유지하면서 `?judge=1`에 영문 입력 흐름과 영문 핵심 브리프를 제공한다. 영문 브리프는 현재·다음 구간, 부족액, 구간 내 위치, 가장 큰 구성 차이, 안전·현금흐름·구성의 세 검토 렌즈와 기존 네 설명 ID의 영향을 보여 준다. 모델 호출 여부는 노출하지 않는다.

Official context: 2025년 가계금융복지조사의 공개 순자산 구간을 `korea-household-net-worth-bands-2025-v1`으로 별도 제공한다. 구간 안의 개인 위치를 보간하지 않고 10억원 이상은 상단 약 11.8%의 넓은 공개구간으로만 표현한다. 이 값은 L1–L15와 8개 자산군 내부 참고범위의 산정 근거로 변환하지 않는다.

Repeat review: 기본 전후 비교는 React 메모리만 사용한다. 장기 비교가 필요한 사용자는 `wealth-report-snapshot-v1` 파일을 명시적으로 다운로드하고 다시 선택할 수 있다. 파일에는 금융 리포트가 포함되므로 사용자 소유이며, 앱은 이를 localStorage·sessionStorage·쿠키·IndexedDB·텔레메트리에 저장하지 않는다. 256 KiB 상한과 strict 중첩 스키마를 적용한다.

## 2026-07-20 — 네 ID 조합 일관성 검증과 설명순서의 본문 반영

Decision: 네 ID 각각의 문맥 허용목록 검사에 더해 프레이밍–선행 인사이트–연결의 의미 호환성을 서버에서 검사한다. 정상 스키마라도 조합이 모순되면 전체 fallback을 반환한다. Responses API가 `completed`가 아니거나 parsed output이 없을 때도 동일하다.

User-facing effect: `explanationOrderId`는 경로 문장뿐 아니라 세 우선순위 카드에서 진단·조정 가이드·확인 기준 중 무엇을 먼저 읽는지 바꾼다. 금융 사실과 순위는 변하지 않는다. 영문 Judge Mode는 동일 네 ID를 bounded explanation plan으로 시각화하지만 한국어 제품 화면에서는 모델 내부와 실행 provenance를 숨긴다.

## 2026-07-20 — 핵심 브리프·세션 비교와 레벨 경로 v2

Decision: 결과 첫 화면은 현재·다음 구간, 부족액과 구간 위치, 가장 큰 구성 차이, 첫 안전조건, 우선 조정 방향 3개와 설명 연결을 하나의 핵심 브리프로 묶는다. 여덟 자산군 상세표와 방법론은 같은 리포트 안에서 펼쳐 보며, 입력값 수정 후 재분석하면 저장 없이 현재 세션 메모리에서 이전 결과와 레벨·부족액·월 잔여액·주요 구성 격차를 비교한다.

Route policy: `level-route-policy-v2`는 L1–L4의 회복·유동성·부채 방어, L5–L9의 자산 역할 분리·형성·집중 완화, L10–L14의 사업·가계 경계·소유책임·전문관리·승계·거버넌스, L15의 영속성 운영을 구분한다. 세 기간은 체크리스트가 아니라 구조 문제, 영향, 조정 수단과 안전선을 연결한다.

Privacy and compatibility: 이전 리포트는 브라우저 저장소가 아닌 React 세션 메모리에만 존재하며 새 리포트 시작이나 새로고침 시 사라진다. 공개 응답은 아래 설명 오케스트레이션 결정에 따라 `wealth-report-v2`를 사용한다.

## 2026-07-20 — 허용목록 기반 설명 오케스트레이션

Decision: OpenAI는 완성된 금융 계산이나 문장을 생성하지 않고, 최소화된 범주형 신호에서 `framingId`, `leadInsightId`, `explanationOrderId`, `connectionId`를 각각 현재 문맥의 허용목록 안에서 선택한다. 서버는 네 ID를 모두 재검증하고 서버 소유 한국어 문장으로 `interpretation`과 `route`를 구성한다. 일부만 유효하거나 필드가 추가된 응답도 전체 fallback으로 처리한다.

Why: 단일 프레이밍 선택만으로는 사용자에게 어떤 근거를 먼저 읽고 각 진단을 어떻게 연결할지 충분히 개인화하지 못했다. 반대로 모델이 금융 수치나 자유 문장을 생성하면 안전성과 재현성이 약해진다. 여러 개의 제한된 설명 결정을 맡겨 모델의 역할을 실질화하되 계산·안전조건·추천 내용은 결정론적으로 유지한다.

Privacy and safety: 원화 금액, 비율, 레벨, 자산구성 값, 제약 메모와 사용자 문장은 모델 입력에 포함하지 않는다. 안전 중단조건과 낮은 데이터 신뢰도에서는 계획을 고정해 모델을 호출하지 않는다. `store: false`, 낮은 추론 노력, 출력 토큰 제한, 짧은 timeout, 해시된 `safety_identifier`를 유지한다.

Compatibility: 새 required `interpretation` 객체 때문에 공개 응답 버전을 strict `wealth-report-v2`로 올린다. 금융 입력이나 리포트를 저장하지 않으므로 저장 데이터 마이그레이션은 없다. v2 클라이언트는 같은 공유 Zod 계약으로 응답을 검증한다.

## 2026-07-20 — 다음 구간 금액범위와 레벨별 경로를 분리

Decision: `composition-policy-v2`부터 자산군별 원화 참고범위와 참고 하단까지의 차이는 `다음 구간 순자산 하단 + 현재 총부채`를 비교용 총자산으로 삼아 계산한다. 현재 부채가 유지된다는 가정은 화면에 명시한다. L1–L15에는 `level-route-policy-v1`의 서로 다른 목적과 0–3·4–6·7–12개월 경로를 적용하며 L15에는 다음 레벨을 만들지 않는다.

Why: 현재 총자산 비중 차이를 원화로 바꾸는 방식은 다음 구간의 실제 규모와 연결되지 않았다. 또한 모든 레벨에 같은 문구를 재사용하면 순자산 회복, 자산형성, 집중위험 관리, 거버넌스라는 단계별 과제가 구분되지 않는다.

Safety: 월 적자는 독립적인 안전 중단조건이다. 90일 예정액은 지급 가능 여부와 지급 후 필수유출 3개월 안전선까지 함께 검사한다. 안전 중단조건은 구성 조정보다 먼저 노출한다.

Compatibility: 공개 응답 구조는 계속 strict `wealth-report-v1`을 유지하고 `methodology.version`만 `composition-policy-v2`로 올린다. 저장 데이터가 없으므로 마이그레이션은 없으며, 클라이언트는 같은 Zod 계약으로 응답을 검증한다.

## 2026-07-19 — 다음 레벨 기준과 서버 전용 분포 보정

Decision: 구성 비교는 현재 레벨이 아니라 반드시 `nextLevel`의 참고범위를 사용한다. L1–L15는 여섯 개 검토 앵커 사이를 서로 다른 값으로 보간하며, 외부 가구 데이터에서는 공개 백분위 좌표의 간격만 서버 전용 보정값으로 사용한다.

Why: 사용자가 알고 싶은 것은 현재 집단의 반복 패턴이 아니라 바로 윗구간의 구조다. 외부 달러 금액을 원화로 환산하면 한국 사용자에게 거짓 정밀도를 만들므로 금액·출처명·통계 순위는 공개 계약에서 제거한다.

Consequence: 공개 화면은 WealthCopy 내부 참고범위만 표시한다. 원시 외부 금액과 백분위 좌표는 클라이언트 번들·API 응답·모델 입력·저장소에 들어가지 않는다.

## 2026-07-19 — 안전조건과 생활 일정이 경로를 지배

Decision: 유동성 개월 수는 생활비와 부채상환액의 합으로 계산하고, 여러 안전 중단조건은 첫 0–3개월에 동시에 점검한다. 주거·이직·사업·큰 지출은 서로 다른 일정·주의선으로 설명하며, 비긴급 제약 메모도 위험과 우선순위에 반영한다.

Why: 우선순위 번호를 3·6·12개월에 기계적으로 배치하면 긴급 위험을 뒤로 미루고 고객 상황과 연결되지 않는다.

## 2026-07-16 — 행동 추적기에서 종합 자산구조 리포트로 전환

Decision: 공개 결과를 체크 가능한 행동과 완료율이 아니라 `레벨 격차 + 여덟 자산군 구성 비교 + 현금흐름·위험 + 우선순위 3개 + 12개월 경로`로 교체한다.

Why: 사용자가 필요한 것은 완료 표시가 아니라 윗구간의 자산이 어떻게 구성되어 있고, 자신의 현재 구조와 얼마만큼 차이가 나며, 무엇부터 조정해야 하는지에 대한 연결된 설명이다.

Consequence: 레거시 v2 계획 API, 행동 카탈로그, 완료 이력, 진행률과 관련 저장 계약을 제거한다.

## 2026-07-16 — 여덟 자산군을 비교의 공통 언어로 사용

Decision: 현금, 거주 부동산, 상장 금융자산, 연금, 수익형 부동산, 사업·비상장, 대체자산, 기타의 여덟 그룹을 입력·분석·출력 전반에서 동일하게 사용한다.

Why: 총자산만으로는 유동성, 편중, 회수 가능성과 현금흐름의 차이를 설명할 수 없다.

Limit: 다음 구간 참고범위는 `composition-policy-v2`의 내부 설계값이다. 관측 모집단 평균, 공식 백분위, 최적 배분 또는 거래 지시로 표현하지 않는다.

## 2026-07-16 — L1–L15와 다음 구간 격차를 서버가 계산

Decision: 여덟 자산군 합계에서 총부채를 뺀 순자산으로 `krw-net-worth-v1`의 L1–L15를 분류한다. 현재 구간 하단부터 다음 기준까지의 위치와 부족액을 함께 계산하며 L15는 최종 운영 구간으로 유지한다.

Why: 레벨만 표시하면 사용자가 현재 위치와 다음 기준의 거리를 이해할 수 없다.

## 2026-07-16 — 공개 계약을 `/api/v3/report`로 교체

Decision: 새 API는 strict 요청으로 금융 스냅샷을 받고 `wealth-report-v1`을 반환한다. 모든 응답은 no-store이며 브라우저는 금융 입력이나 리포트를 저장하지 않는다.

Why: 과거 단계·행동·완료율 중심 계약은 새 리포트 의미를 표현할 수 없고, 이전 완료 이력과 혼합될 위험이 있다.

Evidence: 요청은 클라이언트 총액·레벨·추가 필드를 거절하고, 응답은 레벨·구성·현금흐름·위험·우선순위·경로·신뢰도·방법론으로만 구성된다.

## 2026-07-16 — OpenAI 역할을 경로 프레이밍 선택으로 제한

Decision: 계산, 안전 중단조건, 구성 격차, 우선순위와 사용자 문장은 서버 정책이 결정한다. OpenAI는 정확한 금액과 레벨을 보지 않고 허용된 프레이밍 ID 하나만 선택한다.

Why: 모델의 표현력을 활용하면서도 숫자 계산, 금융 판단과 공개 카피의 결정권을 서버에 유지하기 위함이다.

Fallback: 모델이 없거나 실패해도 동일한 strict 리포트가 결정론적으로 반환된다.
